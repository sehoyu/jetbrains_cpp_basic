이제 리팩토링된 게임에서 충돌 감지 기능을 복원해 봅시다.

우리는 이미 충돌 감지 함수의 시그니처를 변경했습니다.

```c++
CollisionInfo collisionInfo(const Circle& circle1, const Circle& circle2);
```

이제 이 함수는 두 개의 원형 도형을 상수 참조를 통해 전달받습니다. 
충돌이 발생했는지를 나타내는 불리언 플래그 대신,
`CollisionInfo` 구조체를 반환합니다:

```c++
struct CollisionInfo {
    bool collide;
    float distance;
};
```

이 설계는 `collisionInfo` 함수 내부에서 두 객체 간 잠재적인 충돌에 관한 다양한 정보를 계산할 수 있는 유연성을 제공합니다.
또한, 이 정보로 무엇을 할지 결정할 수 있는 기회를 우리 게임의 다른 모듈에 제공합니다.
현재로서는, 충돌이 발생했는지를 나타내는 불리언 플래그 `collide`와
두 객체 사이의 계산된 `distance`를 저장할 것입니다. 하지만,
추후 게임에서 새로운 기능을 구현하는 데 필요한 추가 정보를 이 구조체에 확장하여 포함할 수 있는 기회가 제공될 것입니다.

왜 `CollisionInfo`를 이번 강의에서 배운 고급 객체 대신 구조체로 모델링하기로 결정했는지 궁금할 수 있습니다.
사실, 다음과 같은 요점을 설명하기 위해 의도적으로 그렇게 설계했습니다.
`struct`로 선언된 구조체와 `class`로 선언된 객체/클래스는 C++에서 서로 배타적인 개념이 아니며,
종종 두 개념의 인스턴스가 동일한 코드베이스에서 함께 사용됩니다.

* 객체는 데이터를 (필드)와 동작을 (메서드) 묶는 데 사용됩니다.
  객체는 _캡슐화(encapsulation)_와 _다형성(polymorphism)_을 제공합니다.
  객체의 상태는 클래스 멤버의 가시성을 신중히 제어함으로써 유지되는 여러 불변 조건을 충족할 수 있습니다.

* 구조체는 단순 데이터 컨테이너로 사용됩니다.
  구조체는 예측 가능한 메모리 레이아웃과 예측 가능한 동작을 가지고 있습니다 —
  즉, 런타임에 호출되는 가상 메서드가 없습니다.

C++에서는 구조체를 [_POD 타입_](https://en.wikipedia.org/wiki/Passive_data_structure)으로도 종종 언급하는데,
POD는 _Plain Old Data_를 의미합니다.

<div class="hint">

기술적으로 C++에서는 `class`와 `struct` 키워드 사이에 큰 차이가 없습니다.
예를 들어, `struct` 키워드를 사용하여 클래스를 선언하거나 그 반대로 선언할 수도 있습니다.  
유일한 실질적 차이점은 다음과 같습니다:
* `struct`의 멤버는 기본적으로 `public` 가시성을 가지고 있습니다.
* `class`의 멤버는 기본적으로 `private` 가시성을 가지고 있습니다.

하지만, 대부분의 C++ 개발자들 사이에서는 OOP(객체지향 프로그래밍) 용어로 실제 클래스를 선언할 때는 `class` 키워드를 사용하고,
POD 타입을 선언할 때는 `struct` 키워드를 사용하는 것이 일반적인 관례입니다.

</div>

이제 새로 개편된 충돌 감지 기능을 활용해
소모 가능한(consumable) 객체의 동작을 다시 구현하는 것이 이번 과제입니다.
- 다른 객체와 충돌하면, 소모 가능 객체는 상태를 `DESTROYED`로 변경해야 합니다.
- 다른 객체가 소모 가능 객체에 접근 중인 경우, 상태를 `WARNED`로 변경해야 합니다.
  이는 다른 객체와 소모 가능 객체 간 거리가 `C * r`보다 작을 때 발생해야 합니다. 여기서:
  - `r`은 소모 가능 객체의 반경입니다.
  - `C`는 특별한 승수 상수입니다.

```c++
const float CONSUMABLE_WARNED_MULTIPLIER = 6.0f;
```

이를 구현하기 위해, `ConsumableObject` 클래스의 `onCollision` 메서드를 구현하세요.
이 메서드는 `Scene` 클래스에서 주기적으로 호출되며,
다른 객체와의 잠재적인 충돌 상황을 객체에 알립니다.
이 함수는 첫 번째 인수로 다른 객체를 받으며,
두 번째로 객체 간 거리와 실제로 충돌했는지 여부에 대한 정보를 담고 있는 `CollisionInfo` 구조체를 받습니다.
이 정보를 어떻게 활용할지는 메서드 구현에 달려 있습니다.

또한, 소모 가능 객체가 `WARNED` 상태가 되면,
다른 객체가 주변을 떠났을 때 최종적으로 다시 `NORMAL` 상태로 변경되어야 한다는 점을 유념하세요.
이는 `update` 메서드의 구현을 수정하는 것으로 달성할 수 있습니다.
이 메서드는 역시 `Scene` 클래스에서 주기적으로 호출되어 객체가 내부 상태를 업데이트할 수 있도록 합니다.
이 함수는 단일 인수로 마지막 업데이트 이후 경과된 시간을 받지만,
이번 과제에서는 이 정보가 필요하지 않습니다.
파괴되지 않은(`DESTROYED` 상태가 아닌) 소모 가능 객체의 상태를 `NORMAL`로 재설정하세요.
근처에 다른 객체가 있다면, 이는 다시 `onCollision` 호출 동안 감지될 것입니다.
그렇지 않을 경우, 소모 가능 객체는 `NORMAL` 상태를 유지하게 됩니다.
