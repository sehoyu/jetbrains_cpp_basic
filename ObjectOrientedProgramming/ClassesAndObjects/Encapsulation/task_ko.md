이제 게임의 또 다른 중요한 클래스인 `Scene` 클래스에 대해 살펴보겠습니다.  
게임 씬 객체는 사용자 입력 처리, 게임 객체 유지 및 업데이트 관리,  
창(윈도우)에 그래픽을 그리는 작업, 그리고 기타 활동들을 담당합니다.  
즉, 이 클래스는 우리 간단한 게임 엔진의 핵심 역할을 합니다.

`Scene` 클래스 선언을 살펴보겠습니다.  
이 클래스는 정규 메서드와 가상 메서드뿐 아니라 몇 가지 데이터 필드도 포함하며,  
상당히 다채로움을 자랑합니다.  
이 메서드들의 목적에 대한 자세한 내용은 해당 문서를 참조하십시오.

`Scene` 클래스는 멤버를 세 섹션으로 나눕니다: `public`, `protected`, `private`.  
이제 이들의 의미를 자세히 알아봅시다!

이 _가시성 수정자(visibility modifiers)_ 를 통해  
클래스는 이 클래스의 객체와 클라이언트의 상호작용을 조정할 수 있습니다:

* `public` 섹션에 선언된 모든 필드와 메서드는  
  외부에서 자유롭게 사용 가능하며 클래스 밖에서도 볼 수 있습니다.
* `protected` 섹션에 선언된 모든 필드와 메서드는  
  클래스 내부 자체 및 서브클래스(자식 클래스) 내에서 볼 수 있고 사용할 수 있습니다. 
* `private` 섹션에 선언된 모든 필드와 메서드는  
  클래스 내부에서만 볼 수 있고 사용할 수 있으며, 다른 곳에서는 볼 수 없습니다. 
* 위 규칙에 유일한 예외는, 특정 클래스가 `friend`로 지정되었을 때이며,  
  이 경우 해당 클래스의 보호된(protected) 및 비공개(private) 멤버에 접근할 수 있습니다.

<div class="hint">

상속의 컨텍스트에서 적용될 때,  
예를 들어 `CircleGameObject` 선언에서 다음과 같이 사용될 때:
```
class CircleGameObject : public GameObject
```
가시성 수정자는 다음과 같은 의미를 가집니다:

* 클래스가 `public` 모드로 상속되면,  
  기반 클래스의 public 멤버는 파생 클래스에서도 public으로,  
  기반 클래스의 protected 멤버는 파생 클래스에서도 protected로 유지됩니다.  
  그러나 기반 클래스의 private 멤버는 파생 클래스에서 직접 접근할 수 없습니다.
* 클래스가 `protected` 모드로 상속되면,  
  기반 클래스의 public 및 protected 멤버는 파생 클래스에서도 protected로 유지되며,  
  기반 클래스의 private 멤버는 파생 클래스에서 직접 접근할 수 없습니다.
* 클래스가 `private` 모드로 상속되면,  
  기반 클래스의 모든 public 및 protected 멤버는 파생 클래스에서 private로 변경되며,  
  기반 클래스의 private 멤버는 파생 클래스에서 직접 접근할 수 없습니다.

</div>

클래스의 공개 필드와 메서드는 _public 인터페이스_ 라고도 불립니다.  
(순수 가상 함수로만 구성되고 상태가 없는 클래스에 사용되는 _interface_라는 용어와 혼동하지 마십시오.)  
클래스의 public 인터페이스는 클래스 객체의 외부에서의  
가시성을 정의하며, 클라이언트 클래스가 접근할 수 있는 필드와 메서드를 나타냅니다.

그렇다면 왜 클래스의 일부 필드나 메서드를 숨길 필요가 있을까요?  
결국 외부적으로 유용하게 사용될 수도 있을 텐데 말입니다.  
그러나 특정 _구현 세부 사항_ 을 숨기면  
객체가 자신의 내부 상태를 완전히 제어할 수 있다는 이점이 있습니다.

이 원칙은 _캡슐화(encapsulation)_ 라고 불립니다.  
캡슐화는 클래스 개발자가 객체 데이터의 _불변 조건(invariants)_ 을 유지하도록 하여,  
해당 클래스의 객체가 항상 유효한 상태를 유지하도록 보장합니다.

이 내용을 `Scene` 클래스의 예제를 통해 설명하겠습니다.  
이 클래스는 장면(scene)에 나타나는 게임 객체들을 보관하는 역할을 합니다.  
`Scene` 클래스가 강제할 수 있는 유용한 불변 조건 중 하나는,  
모든 게임 객체가 장면의 경계 내에 있어야 한다는 점입니다.  
하지만, `GameObject` 클래스가 객체 위치를 변경할 수 있는  
`public` 메서드(예: `setPosition`)를 제공할 경우,  
`Scene` 객체는 이 특성을 보장할 수 있는 수단이 없습니다.  
다른 클래스나 함수가 객체의 위치를 변경하여 장면의 가시 영역 밖으로 이동시킬 수 있기 때문입니다.  
반대로, 객체의 위치를 변경할 수 있는 유일한 방법이  
Scene의 메서드(예: `move` 메서드)를 호출하는 것이라면,  
이 메서드 구현은 추가 작업을 수행하여 객체가 장면 내에 남아있도록 보장할 수 있습니다.  
따라서 객체 필드와 메서드의 가시성을 제어함으로써, 개발자는 다양한 유용한 프로그램 상태 불변 조건을 강제할 수 있습니다.

클래스 불변 조건을 마스터하고 멤버의 가시성을 제어하는 기술은  
경험을 통해 익히게 될 것입니다. 더 복잡한 애플리케이션을 설계하고 개발할수록,  
클래스 및 불변 조건 설계에 대해 더 숙련되게 될 것입니다.

이번 단계의 내용을 강화하기 위해  
`Scene` 클래스의 다음 두 메서드를 구현하십시오.

```c++
void setObjectPosition(GameObject& object, Point2D position);
void move(GameObject& object, Point2D vector);
```

위에서 논의한 `Scene` 클래스의 불변 조건 — 즉,  
장면 내에서 객체가 항상 경계 내에 있어야 한다는 조건을 충족해야 합니다.

이 메서드를 구현하려면,  
`GameObject` 클래스의 해당 메서드와 함께  
`Scene` 클래스의 또 다른 메서드인 `fitInto` 메서드를 사용해야 합니다.  
이 메서드는 객체가 `Scene` 경계 내에 맞도록 위치를 조정합니다.

<div class="hint">

참고로 `Scene` 클래스는 `GameObject` 클래스의 `friend`로 선언됩니다  
(즉, `gobject.hpp` 파일의 클래스 선언 참조).  
따라서 `Scene` 클래스는 비록 `GameObject` 클래스에서 private로 선언되었더라도  
`setPosition` 메서드에 접근할 수 있습니다.

</div>
